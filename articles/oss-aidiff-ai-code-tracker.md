# AIが書いたコードを追跡するOSSを作った話

## 課題: 誰もAIコードを追跡していない

Claude Code、Cursor、GitHub Copilot — 今やコードの書き方が根本から変わりつつある。

GitHub Octoverse 2025によると、**商用リポジトリの約41%がAI生成コードを含む**とされている。さらに別の調査では、AIが生成したコードを含むPRは変更失敗率が**約30%高い**というデータも出ている。

それにもかかわらず、ほとんどのチームは「どのコードがAIに書かれたか」を全く把握していない。

具体的にこんな問題がある:

- **どのファイルがほぼAI生成か** わからない
- AI生成コードが**ちゃんとレビューされているか** 追えない
- チーム全体の**AI依存度のトレンド** が見えない

`git blame` で「誰が書いたか」はわかる。でも「AIが書いたかどうか」はわからない。

この空白を埋めるOSSを作った。

---

## aidiff とは

> `git blame` のAI版

```
aidiff scan          # AIコード率を計算して表示
aidiff status        # 詳細な統計情報
aidiff report        # HTMLレポートを生成
```

pip一発で入る。既存のリポジトリに後付けできる。

- GitHub: https://github.com/techino35/aidiff

```bash
pip install aidiff

cd your-repo
aidiff scan
```

## どう動くか — ヒューリスティック検出の仕組み

aidiffは **3層の検出アプローチ** を組み合わせている。

### 1. 明示的タグ（高確度）

```python
# ai-generated
# @ai
# Generated by Claude
// Co-authored-by: GitHub Copilot
```

これらが含まれる行はスコア1.0（AI確定）として扱う。

### 2. コミットメッセージ解析

`Co-authored-by: Claude`, `feat(ai):` などのパターンをgitログから検出してコミット単位でフラグを立てる。

### 3. ヒューリスティック解析（特徴量ベース）

AIが書くコードには一定のクセがある。以下のパターンをスコアリングする:

| パターン | スコア | 例 |
|---|---|---|
| 冗長な変数名 | +0.4 | `user_authentication_token_validation` |
| 自明なコメント | +0.6 | `# Initialize the result dictionary` |
| プレースホルダーTODO | +0.7 | `# TODO: Add error handling` |
| 長すぎるdocstring | +0.3 | 関数本体より長いdocstring |
| 型ヒント付き関数定義 | +0.2 | `def fn(x: int) -> str:` |

さらに **クラスターブースト** がある。AIっぽい行が5行連続で続いている場合、その周囲のスコアも底上げされる。AIは一度にまとまったコードを書くため、孤立した1行よりも連続したブロックの方が信頼度が高い。

### スコアの解釈

- 0.0〜0.4: ほぼ人間が書いたコード
- 0.4〜0.7: AIの関与が疑われる
- 0.7〜1.0: AIが書いた可能性が高い

## CI/CDとgit hookへの組み込み

### pre-commitフックで自動記録

```bash
aidiff install
# → .git/hooks/pre-commit にフックをインストール
# コミットごとに .aidiff/data.json が更新される
```

### GitHub Actionsで閾値チェック

```bash
aidiff init-action
# → .github/workflows/aigate.yml を生成
```

PRのAIコード率が70%を超えると自動でコメントが付く。コードレビューの観点として「このPRはほぼAI生成」という情報がレビュアーに届く。

CI用のクリーンな出力も対応:

```bash
aidiff check --ci --threshold 70
# AI ratio: 34.7%
# Files: 42
# Lines: 8312
# AI lines: 2884
# AI commits: 17/53
```

## なぜ作ったか

自分自身がClaude Codeをヘビーに使うようになって、「これ、後で見返したとき自分が書いたコードか全然わからなくなるな」と感じたのがきっかけ。

AIコードを排除したいわけではない。むしろ逆で、**AI生成コードを可視化することで、どこに人間のレビューが必要かが明確になる**と思っている。

「AIが書いたから信用できない」ではなく「AIが書いたから重点レビューする」という文化を作りたい。
